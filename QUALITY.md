# QUALITY.md

## 0. 目的

この文書は「変更が完了したと言える条件」を定義する。  
目的は品質保証であり、着手前の広域調査や過剰な計画作成を要求するものではない。

原則:
- 変更は最小差分で行う
- 変更に対して十分な根拠を持つ検証を行う
- 変更していない領域まで無差別に検証対象を広げない
- 高リスク変更では検証強度を上げる

---

## 1. 適用ルール

### 1.1 全変更で必須
すべての変更で、最低限以下を満たすこと。

- ビルドまたは型検査が通る
- 変更箇所に関連する検証が通る
- 差分が目的に対して妥当である
- 無関係な変更が含まれていない
- UTF-8 (no BOM) / LF を逸脱していない
- `git diff` と `git status` が説明可能である
- 実行した検証と未実行の検証が区別され、未実行項目を実施済みとして書いていない

### 1.2 高リスク変更で追加必須
以下の領域に変更がある場合は、対応する追加検証を必須とする。

- Web Locks / 単一タブ排他
- Service Worker / COOP-COEP / 起動導線
- import/export / share / save フロー
- DB schema / `user_version` / migration
- SQLite WASM / OPFS 永続化
- `def_hash` 算出
- song master 検証 / 更新
- `packages/web-app` / `packages/shared` / `packages/db` / `packages/pwa` をまたぐ変更
- CI / デプロイ / 依存更新

### 1.3 局所修正の原則
UI文言修正、i18n修正、局所ロジック修正、軽微な表示修正では、変更箇所に対する最小十分な検証を行えばよい。  
変更していない高リスク領域の検証を機械的に追加しない。

---

## 2. 基本検証

### 2.1 ビルド / 型検査
少なくとも以下のいずれか、またはプロジェクト標準の同等検証が成功すること。

- build
- typecheck
- package単位の型検査

変更内容に応じて、必要な範囲で実施する。

例:
- UIのみ変更: web-app中心の build/typecheck
- shared変更: shared を参照する側も確認
- DB/PWA変更: 該当 package も確認

### 2.2 Lint
Lint対象ファイルに変更がある場合、Lintエラーがないこと。

### 2.3 テスト
既存テストがある領域を変更した場合:
- 関連テストが成功すること

テストがない領域でも:
- 少なくとも手動または局所的な再現確認を行うこと

### 2.4 差分検証
以下を確認すること。

- 目的外変更がない
- 無関係な rename / reorder / format-only diff がない
- 生成物の直接編集がない
- lockfile変更は依存更新意図と一致している
- 文字化け、改行揺れ、全文差し替えが起きていない

### 2.5 Validation surface parity / read-back
以下を確認すること。

- build / test / manual check の範囲が、変更した契約面より狭すぎない
- 未検証の gap が残る場合は、その gap と risk を完了報告で明示する
- PR / merge / close / cleanup など write-back を伴う操作は read-back で結果確認する

### 2.6 Boundary / negative path
ロジック、バリデーション、API、payload、contract、request handling に変更がある場合は、少なくとも以下を満たすこと。

- 正常系を1件以上確認する
- 境界値または異常系を1件以上確認する
- 該当する場合は duplicate、empty、nullability、content-type、size limit、optional field、error path を確認する
- happy path のみで完了扱いにしない

### 2.7 New entrypoint smoke check
新規または変更された runnable entrypoint がある場合は、unit test だけでなく最小の runtime smoke check を行うこと。

例:
- Worker / server / CLI の起動確認
- startup / import route の最小導線確認
- runtime wiring や設定不整合が顕在化する経路の確認

---

## 3. UI / 文言 / i18n 変更の検証

以下に該当する変更では、少なくとも対象画面または対象機能の確認を行う。

- UI表示修正
- 文言修正
- i18n修正
- バリデーションメッセージ修正

確認項目:
- 表示崩れがない
- 意図した文言に変わっている
- 既存の導線を壊していない
- ダーク/ライトや主要状態差分がある場合は必要範囲で確認する
- loading / empty / error / disabled / wrap-truncation 状態に影響する場合は必要範囲で確認する

---

## 4. 単一タブ排他の検証（該当変更時のみ必須）

Web Locks / 単一タブ排他ロジックに変更がある場合は、以下を確認する。

- 二重起動時の排他が成立する
- 既存タブ誘導または競合時の挙動が仕様通りである
- 想定外の無限待機、無限リトライ、誤検知がない
- import 起点や share 起点でも破綻しない

---

## 5. PWA / 起動導線の検証（該当変更時のみ必須）

Service Worker、COOP/COEP、crossOriginIsolated、初回起動、import 起動導線に変更がある場合は確認する。

- 初回起動が成立する
- 更新後の挙動が破綻しない
- 必要な再読込導線が機能する
- import / 通常起動 / 共有起点の導線が仕様通り
- キャッシュ起因の不整合が起きない

---

## 6. 保存 / 互換性の検証（該当変更時のみ必須）

DB schema、`user_version`、migration、SQLite WASM、OPFS、import/export 形式に変更がある場合は確認する。

- 既存データを壊さない
- migration が成立する
- import/export の互換性が意図通りである
- 保存後に再起動しても整合性が保たれる
- 失敗時のエラーハンドリングがある

---

## 7. 共有 / ハッシュ / データ契約の検証（該当変更時のみ必須）

`def_hash`、共有URL、payload、shared型に変更がある場合は確認する。

- 既存仕様との整合が取れている
- 同一入力で期待通りの出力になる
- URL/共有経路で破綻しない
- 契約変更がある場合は関連層すべてが追随している

---

## 8. song master / データ検証の確認（該当変更時のみ必須)

song master の取得、検証、更新、参照ロジックに変更がある場合は確認する。

- 更新検知が意図通り動く
- 不正データ時の失敗挙動が明確
- 既存参照ロジックを壊していない
- 必要なら旧データとの互換性が保たれる

---

## 9. 高リスク変更の追加確認

以下に該当する場合は、通常より強い確認を行う。

- 複数 package をまたぐ変更
- CI / deploy 変更
- 依存更新
- 起動不能やデータ消失の可能性がある変更

追加確認:
- 変更対象の境界を越えて破綻していない
- ロールバック可能性がある
- PR説明に影響範囲が記載されている
- 必要ならドキュメントが更新されている

---

## 10. 完了判定

変更は、以下を満たしたとき完了とする。

- 要求された変更が実装されている
- 必須検証が通っている
- 該当変更時のみ必須の追加検証が必要な場合、それも通っている
- 差分が最小かつ妥当である
- 目的外変更がない
- `git status` がクリーン、または未コミット差分が説明可能である

---

## 11. 禁止事項

- 検証していないのに「完了」と判断すること
- 無関係な不具合修正を同じ差分に混ぜること
- format-only diff を品質改善として扱うこと
- 高リスク変更なのに局所修正と同じ検証で済ませること
- 変更していない領域まで機械的にフル検証を要求すること
- 実行していない build / test / manual check / read-back を実施済みとして記載すること
- 正常系だけを確認して境界値や異常系の確認を省略したまま完了扱いにすること
